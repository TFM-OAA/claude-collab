# COLLABORATION.md — правила совместной работы (Google Drive)

> Проект лежит в общей папке Google Drive, синхронизируется через Google Drive for Desktop. Участники работают в Claude Code, каждый со своего аккаунта. **Эти правила обязательны** — Google Drive НЕ сливает правки, а создаёт конфликт-копии.

## ⚠ Главный риск

Google Drive **не мерджит** одновременные правки. Если два человека редактируют один файл «внахлёст», Drive создаёт копию вида `файл (1).md`. Claude Code про синхронизацию не знает и может писать поверх чужих изменений. Поэтому:

## Правила (соблюдать строго)

1. **Работать ПО ОЧЕРЕДИ, не одновременно.** Один работает — второй ждёт.

2. **LOCK-замок `WORKING_NOW.txt` — АВТОМАТИЧЕСКИЙ** (через хуки Claude Code, см. § «Автоматика LOCK» ниже). От вас требуется только: при предупреждении «занято коллегой» — **не работать**, написать коллеге; в конце — закрыть Claude Code (замок снимется сам) или дать `/unlock`.

3. **Синхронизация до и после.**
   - Перед стартом — дождаться, пока значок Google Drive покажет «всё синхронизировано» (зелёная галка). Иначе будете работать со старой версией.
   - После работы — дождаться полной выгрузки **прежде чем** коллега начнёт.

4. **Не запускать Claude Code параллельно** в этой папке с двух машин.

5. **Если увидели `(1)`-копию** — это конфликт синхронизации. Не игнорировать: сравнить с оригиналом, вручную слить нужное, удалить лишнюю копию. Сообщить коллеге.

## Автоматика LOCK (хуки Claude Code)

Замок `WORKING_NOW.txt` обслуживается автоматически — настроено в `.claude/settings.json` + скрипты `.claude/hooks/*.ps1` (синхронизируются всем участникам через Drive). Работает на Windows PowerShell.

**Что происходит само:**

| Событие | Хук / команда | Действие |
|---|---|---|
| **Старт сессии** Claude Code | `SessionStart` → `check-lock.ps1` | Нет замка → создаёт `WORKING_NOW.txt` с вашим `%USERNAME%`, затем **пауза ~10 с** (ждёт синхронизацию Drive) и перепроверка — не захватил ли кто-то параллельно. Замок ваш → «OK, работайте». Замок чужой → **предупреждение** «занято коллегой X». Рядом конфликт-копия → предупреждение о рассинхроне. |
| **Первая правка** (Edit/Write), если замка нет | `PreToolUse` → `guard-lock.ps1` | **Сам ставит замок на вас** (для «старых» сессий, где старт уже прошёл, либо если замок сняли) — с той же паузой-перепроверкой. Чтение/поиск замок не требуют. |
| **Любая правка** при чужом замке / конфликт-копии | `PreToolUse` → `guard-lock.ps1` | **Блокирует** правку (deny): занято коллегой, либо обнаружена `WORKING_NOW (1).txt` (признак одновременного старта — сведите копии). |
| **Конец сессии** (выход/clear) | `SessionEnd` → `remove-lock.ps1` | Снимает замок — но **только ваш** (по `%USERNAME%`). Чужой не трогает. |
| **Команда `/lock`** | вручную | Ставит замок на вас в уже идущей сессии (та же пауза-перепроверка); чужой замок не трогает. |
| **Команда `/unlock`** | вручную | Снимает ваш замок не дожидаясь выхода. |

**Различение свой/чужой:** по `OWNER=<username>` в первой строке `WORKING_NOW.txt` (Windows-логин). У каждого участника логин свой — этого достаточно.

**Страховка от одновременного старта (двухслойная).** Drive не атомарен: если вы и коллега стартуете в одно окно времени, оба можете «успеть» поставить замок, а Drive сделает конфликт-копию `WORKING_NOW (1).txt`. Поэтому:
- При **создании** замка скрипт ждёт `LOCK_SETTLE_SECONDS` (дефолт **10 с**, задаётся env-переменной для тюнинга) и перечитывает — не появился ли чужой владелец / копия. Если да → захват откатывается в предупреждение.
- На **каждой правке** проверяется наличие конфликт-копий; есть → правки блокируются.
- Это **не настоящий мьютекс**: если синхронизация Drive медленнее паузы, старт может промахнуться — но per-edit проверка поймает гонку позже (как только копия доедет). Правила №3/№4 (ждать зелёную галку, не стартовать параллельно) остаются главным предохранителем.

**Что всё равно делает человек:**
- Видишь предупреждение «занято коллегой X» → **не работай**, напиши коллеге, дождись освобождения.
- Видишь **«LOCK SYNC CONFLICT»** или файл `WORKING_NOW (1).txt` → это одновременный старт. **Не работай**, согласуйся с коллегой, сведи копии вручную (правило №5) и удали лишнюю — только потом продолжай. Скрипты копии **сами не удаляют**.
- В конце — просто закрой Claude Code (замок снимется), либо `/unlock`.
- ⚠ **При краше** Claude Code `SessionEnd` может не сработать → замок «повиснет». Тогда: владелец заходит снова (его старт перезапишет/подтвердит) и даёт `/unlock`; либо повисший `WORKING_NOW.txt` удаляют вручную в проводнике (только по согласованию!).
- ⚠ Замок «мягкий»: предупреждение не запрещает физически *открыть* сессию; жёстко блокируются только Edit/Write. Чтение и просмотр доступны всегда.

**Если меняете сам `.claude/settings.json` или хуки** — делайте это по очереди (как любой файл), иначе Drive создаст конфликт-копию конфига.

## Что НЕ синхронизируется (у каждого своё)

- **`WORKING_NOW.txt`** — рантайм-замок, не часть контента (в `.gitignore`).
- **`.claude/settings.local.json`** — ⚠ ЛОКАЛЬНЫЙ файл разрешений Claude Code. Содержит пути и одноразовые команды конкретного участника (`/c/Users/<имя>/...`). У коллеги username другой — эти разрешения просто не сматчатся (не вредно, но бесполезно). **Не считать общим:** при первом запуске Claude Code заново выдаст разрешения под ваши команды. Не редактировать чужие разрешения.
- Память Claude Code (`~/.claude/...`) — у каждого своя; общий контекст проекта — в `CLAUDE.md`.
- Тяжёлые расходники (большие медиа/датасеты/модели), если есть, лучше держать вне Drive или в `.gitignore` — синхронизация будет быстрее.

## Рекомендация на будущее

Для активной параллельной работы Google Drive — не лучший инструмент (риск конфликт-копий остаётся даже при правилах). Приватный Git-репозиторий (GitHub/GitLab) решил бы это радикально (мердж, история, ветки). Если объём совместной работы вырастет — стоит рассмотреть переход.
